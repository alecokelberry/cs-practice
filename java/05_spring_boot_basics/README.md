# 05 — Spring Boot Basics

Spring Boot is the most widely used Java web framework. It takes the powerful-but-verbose Spring Framework and makes it zero-config: sensible defaults, auto-configuration, and an embedded server so there's nothing to install.

Lessons 5, 6, and 7 build on each other. Read them in order.

---

## What Spring Boot Does

Spring Framework has been around since 2003. Its core idea is **Inversion of Control (IoC)**: instead of your code creating its dependencies, a container creates them and hands them to you. Spring Boot (2014) wraps Spring with:

- **Auto-configuration** — detects what's on the classpath and configures it automatically
- **Embedded server** — Tomcat is bundled; no separate install needed
- **Starter dependencies** — `spring-boot-starter-web` pulls in everything needed for a web app
- **`application.properties`** — one file to configure everything

## The Entry Point

```java
@SpringBootApplication
public class DemoApplication {
    public static void main(String[] args) {
        SpringApplication.run(DemoApplication.class, args);
    }
}
```

`@SpringBootApplication` is a shortcut for three annotations:

| Annotation | What It Does |
|------------|-------------|
| `@SpringBootConfiguration` | Marks this as a configuration source |
| `@EnableAutoConfiguration` | Activates auto-configuration |
| `@ComponentScan` | Scans this package and sub-packages for Spring-managed classes |

Spring scans for components starting from this class's package. Put your code **in the same package or a sub-package** — otherwise Spring won't find it.

## The Spring Container and Beans

The **Application Context** is Spring's container. It holds all the objects it manages, called **beans**. When the app starts, Spring creates beans, wires their dependencies, and hands them out.

### Declaring Beans with `@Component`

Mark a class with `@Component` (or one of its specializations) and Spring will create an instance and manage it.

```java
@Component
public class GreetingService {
    public String greet(String name) {
        return "Hello, " + name + "!";
    }
}
```

Specialized `@Component` annotations (prefer these — they communicate intent):

| Annotation | Layer | Use |
|------------|-------|-----|
| `@Controller` / `@RestController` | Web | HTTP request handlers |
| `@Service` | Business | Business logic |
| `@Repository` | Data | Database access |
| `@Component` | Any | Generic bean |

## Dependency Injection

Dependency Injection (DI) is the mechanism by which Spring provides beans their dependencies. Instead of creating dependencies yourself, you declare what you need and Spring supplies it.

### Constructor Injection (preferred)

```java
@Service
public class OrderService {
    private final PaymentService paymentService;
    private final InventoryService inventoryService;

    // Spring detects the single constructor and injects automatically
    // @Autowired is optional when there's exactly one constructor
    public OrderService(PaymentService paymentService, InventoryService inventoryService) {
        this.paymentService = paymentService;
        this.inventoryService = inventoryService;
    }

    public void placeOrder(Order order) {
        inventoryService.reserve(order);
        paymentService.charge(order);
    }
}
```

Constructor injection is the modern best practice. It makes dependencies explicit and allows fields to be `final`, making the class immutable.

### `@Autowired` on Fields (older style — avoid)

```java
@Service
public class OrderService {
    @Autowired  // Spring injects this field directly
    private PaymentService paymentService; // can't be final, harder to test
}
```

Field injection works but makes testing harder (you can't set the field without reflection) and hides dependencies in the class body.

### `@Bean` in Configuration Classes

When you need to create a bean from a third-party class (one you can't annotate), use `@Bean` in a `@Configuration` class:

```java
@Configuration
public class AppConfig {
    @Bean
    public ObjectMapper objectMapper() {
        return new ObjectMapper()
            .configure(DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES, false);
    }
}
```

Spring calls this method once and registers the return value as a bean. Other beans can then inject `ObjectMapper`.

## `application.properties`

```properties
# Server
server.port=8080

# App-level custom property
app.api-key=my-secret-key
app.max-retries=3

# Database (lesson 7)
spring.datasource.url=jdbc:postgresql://localhost:5432/mydb
spring.datasource.username=postgres
spring.datasource.password=secret
spring.jpa.hibernate.ddl-auto=update
```

### Reading Custom Properties

```java
@Component
public class AppProperties {
    @Value("${app.api-key}")
    private String apiKey;

    @Value("${app.max-retries:5}") // 5 is the default if property is missing
    private int maxRetries;
}
```

Or group related properties with `@ConfigurationProperties`:

```java
@ConfigurationProperties(prefix = "app")
@Component
public class AppConfig {
    private String apiKey;
    private int maxRetries;

    // getters and setters (or use a record in Spring Boot 3)
}
```

## Profiles

Profiles let you have different configurations for different environments (dev, prod, test).

```properties
# application.properties — active profile
spring.profiles.active=dev
```

```properties
# application-dev.properties
spring.datasource.url=jdbc:h2:mem:testdb

# application-prod.properties
spring.datasource.url=jdbc:postgresql://prod-db/mydb
```

```java
@Component
@Profile("dev")
public class MockEmailService implements EmailService {
    // used only in dev profile
}
```

## Common Pitfalls

**Component not found**
If Spring says it can't find a bean, the class is likely outside the component scan path. Make sure your class is in the same package as `@SpringBootApplication` or a sub-package.

**Circular dependency**
```
A depends on B, B depends on A → Spring can't create either
```
Circular dependencies usually indicate a design problem. Refactor so one depends on an interface, or extract a third service that both use.

**Using `new` instead of injection**
```java
@Service
public class OrderService {
    // Bug: Spring doesn't manage this — it's a plain object, not a bean
    private final PaymentService payment = new PaymentService();
}
```
Always inject beans; never instantiate them with `new` inside a Spring-managed class.
